Hệ thống có khỏe không?

Đo bằng số: metrics × logs × traces · SLO · burn rate alert.

3 trụ cột

Metrics · Logs · Traces

Observability đứng trên 3 loại dữ liệu (telemetry) mà hệ thống phát ra. Mỗi loại là một góc nhìn riêng.

📈

Metrics

Những con số theo thời gian

Định nghĩa: dữ liệu dạng số, đo lặp lại đều đặn theo thời gian (time-series).

  • Tổng hợp được (đếm, trung bình, %) → nhẹ, lưu lâu
  • Trả lời: "CÓ vấn đề không?" (vd tỉ lệ lỗi, p95 latency)
  • Công cụ: Prometheus
📋

Logs

Nhật ký từng sự việc

Định nghĩa: bản ghi văn bản của từng sự kiện rời rạc, kèm mốc thời gian.

  • Chi tiết nhất nhưng nặng, khó tổng hợp
  • Trả lời: "Lỗi CỤ THỂ là gì?" (vd "10:05 DB timeout")
  • Công cụ: Loki
🔗

Traces

Đường đi của một request

Định nghĩa: hành trình 1 request đi xuyên nhiều service, chia thành các span (mỗi span = 1 chặng, có thời lượng).

  • Thấy độ trễ & lỗi ở từng chặng
  • Trả lời: "Chậm/lỗi ở BƯỚC nào?"
  • Công cụ: Jaeger

Metrics = CÓ lỗi · Logs = lỗi GÌ · Traces = lỗi Ở ĐÂU. Bài thực hành W9 tập trung vào Metrics.

02 — Observability
Mấu chốt · ví dụ thật

Nối 3 trụ cột bằng trace_id

Mỗi request được gắn một mã trace_id duy nhất; mã đó xuất hiện trong cả metric, log và trace.

# Câu chuyện debug 1 lỗi 500 — đi từ metric → log → trace, nhờ cùng trace_id=abc [1] METRIC : 10:05 đồ thị lỗi 500 nhảy vọt → click vào điểm đó, thấy trace_id=abc [2] LOG : lọc Loki theo trace_id=abc → "ERROR: DB connection timeout" [3] TRACE : mở Jaeger trace abc → span api → DB mất 5 giây (tắc ở đây!)

Không có trace_id chung: bạn thấy "có lỗi" (metric) nhưng không biết log nào, trace nào là của request đó → mò kim đáy bể. Có trace_id: 3 cú click từ "lỗi" tới "tắc ở DB 5s".

02 — Observability
Bộ công cụ

Prometheus · Loki · Jaeger · Grafana

App phát tín hiệu → các "kho" lưu lại → Grafana là màn hình xem chung.

Prometheus — kho metrics

Lưu các con số theo thời gian. Điểm cần nhớ: nó tự đi gọi trang /metrics của app mỗi ~15s (gọi là scrape, mô hình "PULL") — app không phải đẩy đi.

Loki — kho logs

Lưu các dòng log, tìm kiếm theo nhãn/thời gian (rẻ hơn full-text).

Jaeger — kho traces

Lưu hành trình request (cây span), tìm theo trace_id hoặc service.

Grafana — màn hình chung

Vẽ dashboard từ cả 3 kho; từ một điểm trên đồ thị có thể bấm nhảy sang log/trace tương ứng.

02 — Observability
Khái niệm lõi

SLA · SLO · SLI

Ba lớp cam kết chất lượng — từ hợp đồng với khách (SLA) → mục tiêu nội bộ (SLO) → con số đo thực tế (SLI).

SLA — Service Level Agreement

Hợp đồng giữa nhà cung cấp & khách hàng: cam kết uptime + phạt nếu vi phạm. (Hướng ra ngoài, có ràng buộc pháp lý/tiền.)

Ví dụ: "AWS cam kết EC2 uptime 99.99%/tháng. Dưới mức đó → hoàn tiền (credit) vào tài khoản bạn."

SLO — Service Level Objective

Mục tiêu nội bộ team đặt cho chất lượng — cụ thể hơn, hướng kỹ thuật. Thường chặt hơn SLA (để còn biên an toàn).

Ví dụ: "Checkout API phải trả lời < 200ms cho 99.5% request trong mọi cửa sổ 30 ngày."

SLI — Service Level Indicator

Con số ĐO thực tế của chất lượng, từ góc người dùng. SLO là mục tiêu; SLI là điểm thực sự đạt được.

Ví dụ: success rate = req không lỗi / tổng req → tháng này 99.7% (≥ SLO 99.5% → đạt).

Nhớ: SLA = hứa với khách (vi phạm thì đền) · SLO = mục tiêu mình tự đặt (chặt hơn SLA) · SLI = số đo thực tế để biết có đạt SLO không. ⚠️ SLI phải đo trải nghiệm user (request thành công, đủ nhanh) — KHÔNG phải "CPU < 80%".

02 — Observability
Khái niệm lõi

Error budget — "ngân sách được phép hỏng"

SLO 99.5% nghĩa là bạn được phép hỏng 0.5%. Tính ra thời gian cụ thể:

# SLO availability = 99.5% trên 30 ngày Tổng thời gian 1 tháng = 30 × 24 × 60 = 43.200 phút Error budget = 0.5% × 43.200 = 216 phút/tháng (~3.6 giờ)

Ý nghĩa (như ngân sách tiêu xài): mỗi tháng có "quota" 216 phút được phép lỗi/down. Còn quota → cứ deploy tính năng mới thoải mái. Hết quota → ngừng mạo hiểm, dồn sức làm ổn định. → SLO biến tranh cãi "deploy hay không" thành quyết định bằng số.

02 — Observability
Khái niệm lõi · cảnh báo

Burn rate — "đang tiêu quota nhanh cỡ nào?"

# Burn rate = mức lỗi hiện tại / mức được phép (0.5%) Lỗi 0.5% (đúng quota) → burn rate = 1 → quota cạn đúng sau 30 ngày Lỗi 5% (gấp 10 lần) → burn rate = 10 → quota 30 ngày cháy trong 3 ngày! Lỗi 7.2% (gấp 14.4) → burn rate = 14.4 → cháy trong ~2 ngày → cực gấp

Burn rate càng cao = càng phải báo gấp. Đừng dùng "alert khi lỗi > 0.5%": quá nhạy (1 phút lỗi nhỏ cũng kêu → báo giả) hoặc quá chậm (rò rỉ âm ỉ không ai biết). → Dùng burn rate qua nhiều cửa sổ thời gian (slide sau).

02 — Observability
Cảnh báo thông minh

Multi-window burn rate alert

Alert nhìn burn rate trên 2 cửa sổ thời gian khác nhau — mỗi cửa sổ bắt một loại sự cố: fast 1h × 5min · slow 6h × 30min.

fast — 1h × 5min

Cửa sổ ngắn → bắt sự cố gấp (incident đang đốt error budget cực nhanh).

→ báo trong vài phút.

slow — 6h × 30min

Cửa sổ dài → bắt rò rỉ chậm (slow leak, lỗi âm ỉ nhiều giờ).

→ báo sớm trước khi cháy hết budget.

Vì sao cần 2 cửa sổ: nếu chỉ 1 cửa sổ thì hoặc quá nhạy (1 phút lỗi nhỏ cũng kêu → báo giả), hoặc quá chậm (rò rỉ âm ỉ không ai biết). Ghép fast (bắt cháy lớn ngay) + slow (bắt rò rỉ âm ỉ) → vừa nhạy, vừa không bỏ sót.

03
Progressive Delivery (Canary)

Ra bản mới từ từ & tự bảo vệ

Thả traffic dần · để metric tự chấm khỏe/bệnh · tệ thì tự abort, không cần người canh.

Problem

Deploy "một phát 100%" rất rủi ro

Bản mới lỗi → toàn bộ user dính ngay.

Ảnh hưởng toàn bộ

Đổi 100% traffic sang bản mới — nếu nó lỗi, mọi user gặp lỗi cùng lúc.

Biết muộn

30 phút sau, khách than phiền, mới phát hiện. Không có "vùng đệm".

Rollback tay

Người phải nhìn dashboard, tự quyết, tự rollback — chậm & căng thẳng.

Progressive delivery (canary) = ra bản mới từng phần nhỏ, để metric tự chấm, tệ thì tự quay về bản cũ trước khi ảnh hưởng số đông.

1% hay 10%? (trade-off)

1%: an toàn hơn (ít user dính nếu lỗi) nhưng metric ít dữ liệu → chấm chậm/khó tin.

10%: đủ dữ liệu chấm nhanh, nhưng lỗi thì 10% user dính.

Quan sát đủ lâu

Mỗi bước cần pause đủ lâu để metric đủ tin rồi mới tăng %. Vội tăng = quyết định trên quá ít dữ liệu.

03 — Canary
Công cụ

Argo Rollouts — thay Deployment bằng Rollout

Rollout là CRD giống Deployment nhưng biết "thả dần + chấm + abort".

kind: Rollout spec: strategy: canary: steps: - setWeight: 10 # 10% traffic sang bản mới - pause: { duration: 2m } # quan sát 2 phút - setWeight: 50 - pause: { duration: 2m } - setWeight: 100 # ok hết -> full

setWeight / pause

setWeight: đặt % traffic cho bản mới. pause: dừng lại quan sát (có thời hạn, hoặc chờ duyệt tay).

Vẫn là GitOps

File Rollout nằm trong Git → ArgoCD sync. Deploy bản mới = đổi image trong Git → push.

03 — Canary
Khái niệm lõi

AnalysisTemplate — "luật tự chấm" bằng metric

Argo Rollouts query Prometheus ở mỗi bước; metric tệ → tự abort & rollback.

kind: AnalysisTemplate # gắn vào Rollout, chạy nền lúc canary spec: metrics: - name: success-rate interval: 30s successCondition: result >= 0.95 # >=95% thành công thì coi là KHỎE failureLimit: 3 # tệ 3 lần liên tiếp -> ABORT provider: prometheus: address: http://prometheus:9090 query: sum(rate(req{status!~"5.."}[2m])) / sum(rate(req[2m]))

Mấu chốt: đây là chỗ nối D2 vào D3 — canary dùng chính metric/SLO để quyết khỏe/bệnh. Nâng cao: cho canary abort theo burn-rate alert (không chỉ success rate tức thời).

03 — Canary
Kết quả

2 kịch bản — không cần người canh

✓ Good run (bản tốt)

10% → quan sát ✓ → 50% → ✓ → 100%

Metric luôn ≥ 95% → canary tự lên dần đến 100% (~20–25 phút).

✗ Bad run (bản mới lỗi)

10% → metric tụt → ABORT → về bản cũ

AnalysisTemplate thấy success rate < 95% → tự abort < 3 phút, rollback, chỉ 10% user từng dính.

Cả 2 ca tự động: không ai ngồi nhìn dashboard, không ai bấm rollback tay. Đó là "tự bảo vệ".

Cài Prometheus + Argo Rollouts — qua GitOps

Không cài tay. Thêm 2 file Application vào argocd/apps/ → git push → root tự cài (đúng app-of-apps buổi sáng).

▶ tự tạo 2 file Application (Helm) trong argocd/apps/
# file: argocd/apps/kube-prometheus-stack.yaml apiVersion: argoproj.io/v1alpha1 kind: Application metadata: { name: kube-prometheus-stack, namespace: argocd } spec: source: repoURL: https://prometheus-community.github.io/helm-charts chart: kube-prometheus-stack targetRevision: 65.1.1 helm: { values: | # repo: + ruleSelector, grafana adminPassword prometheus: { prometheusSpec: { serviceMonitorSelectorNilUsesHelmValues: false } } } destination: { server: https://kubernetes.default.svc, namespace: monitoring } syncPolicy: { automated: { prune: true, selfHeal: true }, syncOptions: [CreateNamespace=true, ServerSideApply=true] } # file: argocd/apps/argo-rollouts.yaml — y hệt, đổi: # chart: argo-rollouts ; targetRevision: 2.37.7 ; namespace: argo-rollouts

Lab 4 · Canary tay

Rollout thả canary — promote / abort bằng tay

Sửa api.yaml đã viết ở Lab 3 (vd VERSION v1→v2) → Rollout không lên 100% ngay, mà dừng ở 25% chờ bạn.

▶ kích 1 bản mới (qua Git)
# sửa k8s-api/api.yaml: VERSION "v1" -> "v2" git commit -am "api v2" && git push # ArgoCD sync -> Rollout bắt đầu canary # theo dõi canary (cần plugin kubectl-argo-rollouts): kubectl argo rollouts get rollout api -n demo --watch
# strategy.canary trong api.yaml: steps: - setWeight: 25 - pause: {} # dừng vô hạn -> chờ người
▶ quyết định bằng tay
# thấy ổn -> cho lên tiếp: kubectl argo rollouts promote api -n demo # thấy tệ -> hủy, về bản cũ: kubectl argo rollouts abort api -n demo
Quan sát: đang pause chỉ 25% pod chạy bản mới; promote → 50%→100%; abort → về bản cũ ngay.
Ở đây người quyết promote/abort. Challenge: thêm AnalysisTemplate query Prometheus để máy tự abort khi metric tệ.
